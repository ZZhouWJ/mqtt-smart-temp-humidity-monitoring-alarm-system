## 第 2 页：MQTT vs TCP 对比

**（时长：1.5分钟）**

**讲解要点：**

1. **TCP 的局限性**
   > 如果直接用 TCP，需要自己实现：心跳机制、消息路由、应用层协议、连接管理。

2. **MQTT 的优势**
   - **发布/订阅模式**：一次发布，多个订阅者同时接收
   - **Topic 主题**：自动消息路由，无需手动分发
   - **QoS 保障**：3 种服务质量，确保消息可靠
   - **Keep Alive**：内置心跳，自动检测连接状态
   - **轻量级**：头部仅 2 字节，适合 IoT 场景

3. **选择 MQTT 的理由总结**
   > MQTT 专为物联网设计，在协议开销、功能完整性和易用性上取得平衡，比 TCP 更适合本项目场景。

**建议 PPT 内容：**

| 对比项 | MQTT | TCP |
|--------|------|-----|
| 通信模式 | 发布/订阅，一对多 | 点对点，一对一 |
| 心跳机制 | 内置 Keep Alive | 需自行实现 |
| 消息路由 | Topic 主题匹配 | 需自行实现 |
| 带宽开销 | 头部仅 2 字节 | 较大 |
| 断线重连 | 支持自动重连 | 需自行实现 |

## 第 7 页：断线自动重连

**（时长：1分钟）**

**讲解要点：**

1. **Python 模拟器：指数退避重连**
   ```python
   reconnect_delay = 1  # 初始延迟 1 秒
   while True:
       time.sleep(reconnect_delay)
       try:
           client.reconnect()
           break
       except:
           reconnect_delay = min(reconnect_delay * 2, 60)
   ```

2. **重连策略要点**
   - 指数退避：避免频繁重连
   - 最大延迟限制：防止无限等待
   - 异常捕获：确保程序稳定

3. **Spring Boot：Paho 自动重连**
   > 后端使用 Eclipse Paho 客户端，配置自动重连，无需手动处理。

## 答辩防守问题速查

### Q1: 滑动平均滤波有什么缺陷？

> 滑动平均滤波可以降低随机噪声，让曲线更平滑，但它存在滞后性。如果真实环境温度快速变化，滤波结果会稍有延迟。本项目的处理方式是：同时保留原始值和滤波值，异常跳变点不进入滑动窗口，避免污染后续曲线。前端同时显示两条曲线，便于观察处理效果。

### Q2: MQTT 的 QoS 为什么选 1？

> QoS 1 表示"至少送达一次"。温湿度数据是高频时序数据，偶尔重复一条可以通过 seq 字段容错处理；但如果使用 QoS 0，消息可能直接丢失。QoS 2 可靠性更高但握手成本更大。因此本项目选择 QoS 1，在可靠性和性能之间取得平衡。

### Q3: 为什么选择 MQTT 而不是 TCP？

> TCP 是更底层的点对点传输协议，若直接使用 TCP，需要自己设计应用层协议、连接维护、多客户端分发和重连逻辑。MQTT 基于 TCP，但提供发布/订阅模型、Topic 解耦、QoS 机制和保活机制，更适合物联网设备数据上报场景。

### Q4: 为什么 JSON 中要加 seq 字段？

> seq 是数据序号，用于丢包检测。后端通过判断 seq 是否连续来发现丢包。例如上一条 seq=1001，下一条 seq=1004，说明中间丢失了 1002 和 1003。这样可以把"丢包检测"从抽象描述落实到代码逻辑中。

### Q5: 为什么没有接入 LLM API 做诊断？

> 本项目以物联网通信、异常处理和实时展示为核心。为保证课堂演示稳定，系统采用本地规则诊断模块，不依赖外部 API Key 和网络服务。诊断规则预先定义，演示时更稳定可控。后续可替换为大模型 API，实现更自然的异常解释。
