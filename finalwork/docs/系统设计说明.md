# 系统设计说明

## 一、项目目标

本项目实现一个基于 MQTT 协议的模拟环境监测物联网系统。系统使用电脑模拟温湿度采集节点，通过 MQTT 上传到 Broker，后端订阅数据并进行滤波、丢包检测、报警判断和诊断建议生成，前端实时展示温湿度曲线和报警记录。

## 二、总体架构

```text
Python 模拟传感器
        ↓ MQTT 发布 JSON
EMQX / 公共 MQTT Broker
        ↓ MQTT 订阅
Spring Boot 后端
        ↓ WebSocket 推送
Vue + ECharts 前端
```

系统对应物联网分层：

| 层级 | 实现 |
|---|---|
| 感知层 | Python 模拟温湿度传感器 |
| 网络层 | MQTT 发布 / 订阅通信 |
| 平台层 | Spring Boot 数据处理与报警 |
| 应用层 | Vue + ECharts 可视化大屏 |

## 三、MQTT 与 TCP 对比

| 对比项 | MQTT | TCP Socket |
|---|---|---|
| 通信模式 | 发布 / 订阅 | 点对点连接 |
| 解耦程度 | 通过 Topic 解耦设备和后端 | 通信双方直接绑定 |
| 消息分发 | Broker 自动分发 | 需要自行实现广播或转发 |
| QoS | 支持 QoS 0/1/2 | 需要应用层自定义可靠性机制 |
| 适用场景 | 物联网设备数据上报 | 底层传输或专用协议 |

本项目选择 MQTT 的原因是：温湿度设备属于典型物联网采集节点，数据量小、上报频繁、设备与服务器需要解耦，MQTT 的轻量发布/订阅机制更加适合。

## 四、数据格式设计

系统采用 JSON 格式：

```json
{
  "deviceId": "env-node-001",
  "seq": 1001,
  "timestamp": "2026-06-15 20:30:00",
  "temperature": 28.6,
  "humidity": 65.2,
  "status": "normal"
}
```

选择 JSON 的原因：可读性强、跨语言解析方便、字段结构清晰、便于后续扩展 PM2.5、光照、烟雾等传感器字段。

## 五、断线自动重连机制

Python 发布端采用指数退避重连：1 秒、2 秒、4 秒、8 秒逐步重试，最高 30 秒，避免 Broker 异常时频繁重连。

Spring Boot 后端使用 Eclipse Paho 客户端，开启 `automaticReconnect=true`，并在 `connectComplete` 回调中重新订阅 Topic。后端首次连接失败时也会在后台线程中执行指数退避重试，适合 Docker Compose 场景。

## 六、异常数据与滤波策略

系统模拟以下异常：

1. 随机白噪声：正常采样值中加入小范围波动；
2. 跳变异常：温度或湿度突然大幅变化；
3. 丢包异常：模拟某次数据不发布；
4. 离线异常：模拟设备短暂停止上报。

处理策略：

- 滑动平均滤波：对最近 N 个有效数据求平均；
- 跳变检测：当前值与上一有效值差值超过阈值则标记为 outlier；
- 异常点隔离：异常点不进入滑动窗口，避免污染平滑曲线；
- 丢包检测：通过 seq 序号是否连续判断；
- 本地诊断：根据异常类型生成维护建议。

## 七、工程化改进

本次改进新增 Docker 一键部署和本地规则诊断模块。Docker 提升部署稳定性；诊断模块增强报警解释能力。时序数据库和 LLM API 作为后续拓展方向，不作为核心演示依赖。
