# 答辩防守问题准备

## 1. 为什么选择 MQTT 而不是 TCP？

**参考答案：**

TCP 是更底层的点对点传输协议，若直接使用 TCP，需要自己实现：

- 应用层协议设计（消息格式、字段定义）
- 心跳保活机制（检测连接是否断开）
- 消息路由分发（多客户端场景）
- 断线重连逻辑

MQTT 基于 TCP，但专为物联网场景优化：

| 特性 | TCP | MQTT |
|------|-----|------|
| 通信模式 | 点对点，一对一 | 发布/订阅，一对多 |
| 心跳机制 | 需自行实现 | 内置 Keep Alive |
| 消息路由 | 需自行实现 | Topic 主题匹配 |
| 带宽开销 | 较大 | 头部仅 2 字节 |
| 适用场景 | 通用 | IoT、移动端 |

**本项目选择 MQTT 的理由：**
1. 发布/订阅模式解耦设备端和后端，便于扩展
2. Topic 机制自动实现消息路由
3. QoS 保障消息可靠传输
4. 协议简单，开源库丰富，开发效率高

## 2. 为什么选择 JSON 而不是自定义格式？

**参考答案：**

本项目使用 JSON 格式设计数据，原因如下：

**JSON 的优势：**
1. **可读性强**：人类可读，便于调试和日志分析
2. **跨语言支持**：Python、Java、JavaScript 原生支持 JSON 序列化/反序列化
3. **自描述性**：字段名即含义，无需额外解析文档
4. **扩展性强**：新增字段（如 deviceName、location）不影响现有解析逻辑
5. **标准化**：业界通用格式，便于与第三方系统集成

**自定义二进制格式的缺点：**
- 体积更小，但牺牲了可读性
- 需要自行设计序列化/反序列化逻辑
- 字段扩展需要考虑版本兼容

**本项目数据量小（每秒几条消息），可读性优先级更高，因此选择 JSON。**

## 3. 为什么 JSON 中要加 seq 字段？

**参考答案：**

seq 是数据序号字段，用于**丢包检测**。

**工作原理：**
```java
// 假设接收到的数据序列
// seq: 1001, 1002, 1003, 1006, 1007...
//                    ↑ 这里跳过了 1004, 1005

if (current.seq != lastSeq + 1) {
    int lostCount = current.seq - lastSeq - 1;
    System.out.println("检测到丢包，数量：" + lostCount);
}
```

**seq 字段的作用：**
1. **丢包检测**：发现传输过程中丢失的数据包
2. **数据完整性验证**：确保没有遗漏采集点
3. **故障定位**：丢包频繁时说明网络不稳定

## 4. 断线自动重连机制是怎么实现的？

**参考答案：**

本项目在发布端和订阅端都实现了断线重连。

**Python 发布端：指数退避重连**
```python
def on_disconnect(client, userdata, rc):
    if rc != 0:
        reconnect_delay = 1  # 初始延迟 1 秒
        max_delay = 60       # 最大延迟 60 秒
        
        while True:
            print(f"等待 {reconnect_delay} 秒后重连...")
            time.sleep(reconnect_delay)
            try:
                client.reconnect()
                print("重连成功")
                break
            except Exception as e:
                print(f"重连失败: {e}")
                # 指数退避：延迟翻倍，上限 60 秒
                reconnect_delay = min(reconnect_delay * 2, max_delay)
```

**重连策略要点：**
| 要点 | 说明 |
|------|------|
| 指数退避 | 避免频繁重连造成服务器压力 |
| 最大延迟限制 | 防止无限等待 |
| 异常捕获 | 确保程序不会崩溃 |
| 日志记录 | 便于排查问题 |

**Spring Boot 后端：使用 Paho 内置重连**
> Eclipse Paho MQTT 客户端默认配置了自动重连，无需手动实现。

## 5. 如何模拟传感器异常数据？

**参考答案：**

本项目模拟三种异常：

**1. 跳变异常（数据突变）**
```python
# 5% 概率触发，大幅偏离正常值
if random.random() < 0.05:
    temperature += random.uniform(-15, 15)
```

**2. 丢包模拟**
```python
# 3% 概率跳过发送
if random.random() < 0.03:
    continue  # 不发送本次数据
```

**3. 离线模拟**
```python
# 每 60 秒模拟一次短暂离线（5秒）
if offline_timer >= 60:
    print("设备离线...")
    client.loop_stop()
    time.sleep(5)  # 离线 5 秒
    client.loop_start()
    offline_timer = 0
```

## 6. 滑动平均滤波有什么缺陷？

**参考答案：**

滑动平均滤波的优点是简单有效，可以平滑随机噪声。**缺点**是：

1. **滞后性**：对快速变化反应迟钝
2. **边缘效应**：窗口滑动时两端数据有延迟

**本项目的解决方案：**

```java
// 1. 同时保留原始值和滤波值
public class SensorData {
    double rawTemperature;      // 原始值
    double filteredTemperature; // 滤波值
    boolean outlier;            // 是否异常点
}

// 2. 异常跳变不进入滤波窗口
if (isJump(current, last)) {
    current.outlier = true;  // 标记为异常
} else {
    // 正常数据才进入滤波窗口
    filtered = slidingAverage(current);
}

// 3. 前端同时显示两条曲线
series: [
    { name: '原始值', data: rawData },
    { name: '滤波值', data: filteredData }
]
```

这样既能平滑噪声，又能保留快速变化的信息。

## 7. 为什么选择 QoS 1 而不是 QoS 0 或 QoS 2？

**参考答案：**

MQTT 有 3 种服务质量级别：

| QoS | 名称 | 保证 | 开销 | 适用场景 |
|-----|------|------|------|----------|
| QoS 0 | 最多一次 | 不保证送达 | 最低 | 不重要数据 |
| QoS 1 | 至少一次 | 可能重复 | 中等 | 重要数据 |
| QoS 2 | 恰好一次 | 严格一次 | 最高 | 关键数据 |

**本项目选择 QoS 1 的原因：**

1. **温湿度数据需要可靠传输**：不能丢失重要告警数据
2. **重复数据可容忍**：通过 seq 字段可以去重
3. **性能适中**：QoS 2 的握手成本不适合高频采集

## 8. 为什么没有真正接入 LLM API？

**参考答案：**

本项目以**物联网通信、异常处理和实时展示**为核心功能。

**本地规则诊断的优点：**
1. **演示稳定**：不依赖外部 API Key 和网络
2. **响应快速**：规则匹配即时响应
3. **可控性强**：诊断结果可预期

**诊断规则示例：**
```java
public String diagnose(AlarmType type) {
    return switch (type) {
        case TEMP_HIGH -> "检查设备散热、环境通风、减少负载";
        case HUMIDITY_HIGH -> "检查除湿设备、窗户密封、通风系统";
        case OFFLINE -> "检查设备电源、网络连接、MQTT Broker 状态";
        default -> "请联系运维人员";
    };
}
```

**后续可扩展：**
> 可将 DiagnosisService 替换为大模型 API，实现更自然的异常解释和智能诊断建议。
